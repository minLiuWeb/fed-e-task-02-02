# 简单题
#### 1.Webpack 的构建流程主要有哪些环节？如果可以请尽可能详尽的描述 Webpack 打包的整个过程。
1. 寻找入口
    - 首先根据指定文件找到打包依据
    - 找到打包的起点即入口(entry)指示 webpack 应该使用哪个模块,来作为构建其内部依赖图的开始。
    - 进入入口后,webpack 会通过递归的方式找出有哪些模块和库是入口（直接和间接）依赖的。
    - 每个依赖项随即被处理,最后输出到称之为 bundles 的文件中
    - 如果没有指定，则默认为src/index.js

2. 出口
    - output 属性告诉 webpack 在哪里输出它所创建的 bundles,以及如何命名这些文件,默认值为 ./dist

3. Module
    - 模块,在 Webpack 里一切皆模块,我理解为每一个文件都是一个模块。Webpack 会从配置的 Entry 开始递归找出所有依赖的模块。

4. Chunk
    - 代码块,一个 Chunk 由多个模块组合而成,用于代码合并与分割

5. Loader
    - loader让webpack 能够去处理那些非 js 文件（webpack 自身只理解 js）
    - loader 可以将所有类型的文件转换为 webpack 能够处理的有效模块,然后你就可以利用 webpack 的打包能力,对它们进行处理。

6. Plugin
    - 插件可以用于执行范围更广的任务。
    - 插件的范围包括,从打包优化和压缩,一直到重新定义环境中的变量。插件接口功能极其强大,可以用来处理各种各样的任务。

执行顺序为：

1. 找到入口
2. 从入口出发找到依赖的模块并调用loader进行加载与处理，并对此模块继续查找所使用的模块进行loader处理，直至所有的模块都是处理后的代码
3. 在上述过程执行过程中，我们的plugin进行监控，可以对代码进行优化和压缩
4. 根据我们的output配置输出资源并写入文件系统





#### 2.Loader 和 Plugin 有哪些不同？请描述一下开发 Loader 和 Plugin 的思路。

不同：

webpack本身只能处理js代码，对于非js代码，loader定义了如何处理其他资源。loader可以将文件从不同的语言（less）转换为js，或者将内联图像转换为data URL。

plugin插件可以用于执行范围更广的任务。插件接口功能极其强大,可以用来处理各种各样的任务。


开发思路：

loader: 首先是loader接收模块资源，然后我们进行处理后返回一段js 

plugin:必须一个函数或一个包含apply方法的对象。然后可以将此方法挂载到webpack的生命周期上。
